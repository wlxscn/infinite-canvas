## ADDED Requirements

### Requirement: 助理最终消息支持 Markdown/GFM 渲染
系统 SHALL 在右侧聊天侧栏中将助理的最终消息按 Markdown 渲染，而不是仅按纯文本保留换行；该最终态渲染 SHALL 支持常见段落、列表、强调、引用、链接、行内代码、代码块以及 GFM 表格。

#### Scenario: 助理最终消息显示结构化列表与代码块
- **WHEN** 助理返回一条已经完成的消息，其中包含 Markdown 列表、强调和 fenced code block
- **THEN** 聊天侧栏将该消息按对应的结构化元素渲染，而不是原样显示 Markdown 标记字符

#### Scenario: 助理最终消息显示 GFM 表格
- **WHEN** 助理返回一条已经完成的消息，其中包含 GFM 表格语法
- **THEN** 聊天侧栏将该消息中的表格渲染为可读的表头和单元格结构

### Requirement: 流式回复阶段保持纯文本打字机效果
系统 SHALL 在助理消息仍处于流式生成阶段时继续使用纯文本打字机展示，而 MUST NOT 在该阶段实时解析未完成的 Markdown 结构。

#### Scenario: 流式中的 Markdown 语法不提前结构化
- **WHEN** 助理正在流式返回一条包含 Markdown 标记的消息，且该消息尚未完成
- **THEN** 聊天侧栏继续以纯文本逐字显示当前内容，而不会在消息完成前把列表、代码块或表格提前渲染成最终结构

#### Scenario: 流式完成后切换为 Markdown 最终态
- **WHEN** 一条流式助理消息完成并被作为最终消息写入当前会话
- **THEN** 聊天侧栏将该消息切换为 Markdown/GFM 最终态渲染
- **AND** 系统不会为同一条消息额外显示一份重复的纯文本流式占位

### Requirement: Markdown 表格与宽内容在侧栏中保持可读
系统 SHALL 让 Markdown 表格和其他超宽正文内容在右侧窄侧栏中保持可读，而 MUST NOT 因宽内容导致聊天气泡或整个侧栏布局被撑破。

#### Scenario: 宽表格在消息内部横向滚动
- **WHEN** 助理最终消息包含超出聊天气泡宽度的 Markdown 表格
- **THEN** 系统在该消息正文内部提供局部横向滚动区域以查看完整表格
- **AND** 聊天气泡与侧栏主布局保持稳定

#### Scenario: 代码块宽度不会破坏侧栏布局
- **WHEN** 助理最终消息包含较长的代码行或宽代码块
- **THEN** 系统允许用户在消息正文内部查看完整代码内容
- **AND** 右侧聊天侧栏与输入区不会因该代码块而超出既有布局边界
