## Context

仓库已经具备“单个项目快照持久化”的基础：

- 前端通过 `resolveProjectId()` 在本地生成或复用稳定 `projectId`
- 启动时先从 `localStorage` 加载项目，再尝试从后端水合当前项目
- 后端已支持 `GET /projects/:projectId` 与 `PUT /projects/:projectId`
- `CanvasProject` 继续承载画布内容、资产、生成任务和项目内聊天会话

但当前系统缺少“项目目录层”：

- 没有画布列表入口
- 没有“新建画布”行为
- 没有项目级标题
- 没有正式的“切换到另一张画布”状态语义

这意味着底层已经具备多个 `projectId` 的地址空间，上层却没有把它表达为产品能力。

本次设计聚焦第一版画布管理，不改变 `CanvasProject` 作为单画布快照的主体地位，而是在其外层补一层“项目摘要 + 项目选择 + 项目切换”的管理能力。

## Goals / Non-Goals

**Goals:**

- 支持从界面新建空白画布项目。
- 支持查看最近画布列表并切换到目标项目。
- 为项目提供独立标题和更新时间等基础元数据。
- 让项目管理能力可在本地优先和后端持久化并存的模式下运行。
- 明确“切换项目”时哪些运行时状态需要重置，哪些持久项目状态应被保留。
- 在不引入协作和分享的情况下，形成完整的单人多画布工作流。

**Non-Goals:**

- 不实现多人协作、presence、CRDT 或冲突合并。
- 不实现项目分享、权限控制、成员管理和公开链接。
- 不实现文件夹、标签、归档、收藏和回收站。
- 不实现复杂搜索、筛选和分页管理中心。
- 不把聊天会话中心提升为跨项目的全局消息系统。

## Decisions

### 1. 保持 `CanvasProject` 作为单画布快照，新增独立的项目摘要模型

决定：

- `CanvasProject` 继续只承载画布文档本体。
- 新增项目摘要模型 `ProjectSummary` 或等价结构，用于列表和管理场景。
- 项目摘要至少包含：
  - `projectId`
  - `title`
  - `createdAt`
  - `updatedAt`
  - 可选 `thumbnailUrl`
  - 可选 `lastOpenedAt`

原因：

- 画布列表并不需要每次拉取完整 `CanvasProject`。
- 当前 `ChatSession.title` 是会话名，不适合复用为画布名。
- 将“项目摘要”和“项目快照”分开，有利于后续演进成项目目录或仪表盘。

备选方案：

- 把项目标题直接塞进 `CanvasProject.chat` 或会话标题中：短期省事，但语义错误，会让列表和编辑逻辑持续耦合。

### 2. 第一版后端项目管理能力最小闭环为“列出 / 创建 / 重命名”

决定：

- 在现有单项目读写接口之外，补充最小项目管理接口：
  - 列出项目摘要
  - 创建新项目
  - 更新项目元数据中的标题
- 删除项目不进入第一版范围。

原因：

- “新建 + 列表 + 重命名”已经足够形成可用的多画布工作流。
- 删除会引入额外确认、恢复、空态和数据安全问题，首版可以避免。

备选方案：

- 只在前端本地维护最近项目列表，不增加后端接口：实现更快，但跨设备时无法形成稳定的画布管理体验。

### 3. 未登录第一版采用“本地最近项目索引 + 后端项目列表可用即优先”的混合策略

决定：

- 前端保留本地最近项目索引，记录当前浏览器访问过的项目标识和最近打开时间。
- 如果后端项目列表可用，则以服务端返回的项目摘要为主。
- 如果后端不可用或未配置，仍允许用户在本机通过本地索引继续切换已访问项目，并新建本地可恢复项目。

原因：

- 这与现有 local-first + backend-preferred 的持久化设计一致。
- 可以避免把“画布管理”绑定到 Supabase 正常可用这一单一前提。

备选方案：

- 纯后端列表：跨设备一致性更好，但会让本地开发、弱网和无后端模式下的画布管理退化为不可用。

### 4. “新建画布”必须显式创建新的项目身份，而不是覆写当前项目

决定：

- 新建画布时生成新的 `projectId`，并进入新的空白项目。
- 当前项目保持不变，不通过清空现有 `CanvasProject` 的方式实现“新建”。
- 新建后应更新 URL，使项目链接可复制、可刷新恢复。

原因：

- “新建”在用户心智上意味着创建新对象，而不是重置当前对象。
- 覆写当前项目会破坏最近项目列表、历史恢复和后端持久化语义。

备选方案：

- 在当前 `projectId` 下清空项目内容：会造成项目身份与项目历史断裂，不适合作为正式项目管理行为。

### 5. 区分“当前项目远端水合”和“切换到另一项目”

决定：

- 远端水合：用于启动时把同一 `projectId` 的远端快照覆盖本地初始值，不进入历史。
- 切换项目：用于从项目 A 进入项目 B，需要重置运行时状态并加载新项目。

切换项目时应重置的运行时状态包括：

- `selectedId`
- `selectedIds`
- `activeGroupId`
- `past` / `future`
- 当前工具栏上下文中只属于前一个项目的临时状态
- 与前一个项目不兼容的流式消息或面板聚焦状态

原因：

- 这两个动作虽然都表现为“替换当前 project”，但产品语义完全不同。
- 如果不显式区分，切换项目后可能带入错误的选择态、撤销栈和导航上下文。

备选方案：

- 统一使用 `replaceProjectNoHistory()`：对单项目水合成立，但对跨项目切换不够完整。

### 6. 项目标题采用显式项目元数据，不自动绑定到聊天标题

决定：

- 新项目使用默认标题，例如“未命名画布”。
- 用户可在项目管理入口显式重命名。
- 聊天会话标题仍只服务于会话列表，不自动反写为项目标题。

原因：

- 一张画布可以有多轮不同主题的聊天，会话标题不稳定。
- 项目标题需要更长期、更稳态的语义。

备选方案：

- 用首条用户消息自动命名项目：可以做为后续增强，但不应替代显式项目标题。

## Data Model Sketch

```text
ProjectSummary
  ├─ projectId
  ├─ title
  ├─ createdAt
  ├─ updatedAt
  ├─ ownerId?
  ├─ thumbnailUrl?
  └─ lastOpenedAt?

CanvasProject
  ├─ version
  ├─ board
  ├─ assets
  ├─ jobs
  └─ chat
```

可接受的后端存储方向：

```text
projects
  ├─ id
  ├─ owner_id
  ├─ title
  ├─ data
  ├─ created_at
  └─ updated_at
```

如果短期不想把 `title` 拆到顶层列，也至少需要保证项目列表接口能稳定返回该元数据，而不是要求前端逐个拉完整快照推导标题。

## UX Flow Sketch

```text
进入应用
  │
  ├─ 解析当前 projectId
  │    └─ 有则加载当前项目
  │
  └─ 打开“画布切换器”
       ├─ 查看最近画布
       ├─ 新建画布
       └─ 选择某张画布
              │
              ▼
        更新 URL(projectId)
              │
              ▼
        重置运行时状态
              │
              ▼
        加载目标 CanvasProject
```

## Risks / Trade-offs

- [项目管理范围容易膨胀成完整文件系统] → 第一版只做最近画布、基础命名和切换，不做文件夹、标签和删除。
- [本地最近项目索引与后端列表可能短暂不一致] → 明确后端可用时以后端为主，本地索引作为离线和降级兜底。
- [项目切换会与现有单项目水合逻辑混淆] → 在状态层显式区分“hydrate current project”和“switch project”两个动作。
- [项目标题来源不明确会引入重复命名逻辑] → 项目标题独立于聊天标题，避免隐式同步。
- [未登录场景下的跨设备一致性有限] → 第一版接受“本地优先，可配合后端增强”的渐进式能力，不把账号体系纳入本次范围。
