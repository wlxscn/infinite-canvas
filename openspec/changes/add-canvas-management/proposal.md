## Why

当前系统已经支持基于 `projectId` 的单个画布项目加载、保存和刷新恢复，但产品层仍然停留在“当前这一张画布”的模型里。用户虽然可以在技术上通过 URL 指向不同 `projectId`，却无法在界面中直接完成这些高频动作：

- 新建一张空白画布
- 查看最近使用过的画布
- 在多张画布之间切换
- 为画布提供稳定的名称，而不是把会话标题误当成画布标题

这会导致两个问题同时存在：

- 已经具备的后端项目持久化能力无法被自然使用，系统缺少“项目目录层”
- “画布管理”相关心智被隐含在 URL 和本地缓存里，用户无法形成可见、可操作的多画布工作流

本变更的目标，是在不引入协作、分享、权限和复杂目录系统的前提下，为 Infinite Canvas 增加第一版画布管理能力，让“新建画布”和“切换画布”成为正式产品能力。

## What Changes

- 新增画布管理能力，使用户可以从界面新建空白画布，而不必依赖手工修改 URL 或清理本地缓存。
- 新增画布目录/最近画布视图或等价入口，让用户可以看到最近访问或可恢复的画布列表，并切换到目标画布。
- 为画布引入独立于聊天会话标题的项目元数据，至少包含项目标题、创建时间、更新时间和标识符。
- 在后端项目持久化能力之上补充“项目列表 / 项目创建 / 项目重命名”的管理接口或等价编排能力。
- 明确区分“水合当前画布”和“切换到另一张画布”的语义，避免把跨项目切换误当成单项目的远端覆盖。
- 保持现有单项目快照结构 `CanvasProject` 继续作为画布文档主体，不把本次改动扩大为文档模型重构。
- 保持第一版边界：
  - 不实现多人协作
  - 不实现项目分享与权限 UI
  - 不实现文件夹、标签、归档和回收站
  - 不实现复杂冲突合并
  - 不要求引入登录后才能使用的完整账号体系

## Capabilities

### New Capabilities

- `canvas-project-management`: 定义新建画布、列出最近画布、切换画布、基础重命名和项目级元数据管理的行为。

### Modified Capabilities

- `ai-design-canvas`: 画布工作区将从“单当前项目编辑”扩展为“可在多个项目之间切换的编辑体验”，并明确跨项目切换时的运行时状态重置边界。
- `project-backend-persistence`: 项目持久化将从单项目读写扩展为支持项目管理入口所需的最小元数据与列表能力。

## Impact

- 影响 `apps/web`：
  - `WorkspaceHeader`、应用初始化流程和项目切换路径需要新增项目标题、最近画布入口和新建画布入口。
  - 需要补充项目目录状态、切换语义和本地最近项目索引。
  - 需要在切换项目时清理或重置只属于当前运行时的选择状态、历史栈和活动导航上下文。
- 影响 `apps/agent-api`：
  - 需要在现有 `GET /projects/:projectId`、`PUT /projects/:projectId` 之外，增加项目列表、项目创建和项目元数据更新相关能力。
  - 需要明确后端返回的项目管理元数据和排序策略。
- 影响 `packages/shared`：
  - 需要增加项目摘要、项目列表、项目创建和项目更新的共享 API 合同。
- 影响 Supabase schema：
  - 需要让 `projects` 表或等价存储结构支持项目级 metadata，而不只是一整份 `data jsonb` 快照。
- 不影响范围：
  - 不修改画布节点模型
  - 不修改聊天 agent 的核心决策逻辑
  - 不引入实时同步协议
